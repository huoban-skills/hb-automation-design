# 出方案前自检清单（preflight）

输出「方案确认」前逐区走查本清单（**内部动作，结论不输出给用户**）：对每个命中项，在方案节点清单里指认出对应节点——指认不到说明方案缺节点，先补方案再输出。方案里不出现「自检」段落，自检的体现就是方案本身完整。

> **收录边界**：本清单只收**方案层遗漏**——产出的 JSON 技术上合法、`validate`/`verify` 查不出来的业务缺口（如漏明细表、缺数组节点）。JSON 写法、字段壳层、回显等**实现层避坑**不要加进来，它们由 `hac automation docs show` / `validate` / `verify` + 编辑器回显检查兜底，集中在 design-principles 第二区。

## 区1：创建带明细的主单据（订单发货 / 入库 / 采购 / 领料……只要主单通过关系字段关联明细表就命中）

- **明细表没漏**：主单关联的明细表已作为子表在「打开数据创建页面」里预填？默认必须预填——要省略，必须先 `hac table get-table` 核对确认无法预填并在方案写明；"字段列表没看到子表字段"不是省略理由。
- **子表有数组来源**：「打开数据创建页面」**前面有产出数组的节点**（「获取多条数据」/ 关联字段带出的多值等）？方案里写明"子表每行 ← 哪个节点的数组"。没有数组源 = 子表一行都填不进去，先补取数组节点。
  - 正例（订单发货）：按钮 →「获取多条数据」查待发货明细（**产出数组**）→「打开数据创建页面」建发货单，主表填订单信息、子表用上一步数组预填明细行。
- **交互而非后台**：创建主业务单据默认「打开数据创建页面」；要走「后台创建数据」必须同时满足后台准入四条件（用户明确要求纯后台 / 字段完全确定 / 明细无需调整 / 不依赖预填子表，见 design-principles 第一区）并写明理由。

## 区2：循环

- 模式选对：每条数据只执行一次 → 数组循环；同条数据多次执行 → 条件循环。
- 没有循环套循环；需要嵌套就拆「执行调用触发」子流程。
- 循环体内没有交互节点。

## 区3：数值判断 / 分支 / 数量分配

- 已查过现成计算字段（本表原生 + 关联附加，核对 `from_relation_field` 口径），优先用计算字段，不先用「数据统计」/「设置变量」重算。

## 区4：第三方接口（api_call）

- 方案写明：失败怎么办、返回结构是否稳定、后续节点读什么字段。

## 区5：结束与失败反馈

- 有且只有一个「结束」；中断路径用「提前退出」，需要用户看到原因时 `exit + dialog`。

## 区6：通用

- 真的需要这个自动化吗？有没有更简单的入口（如按钮显隐条件就能拦住的校验）？
- 字段归属核对过：本表原生 / 关联附加 / 系统字段，没有未查表就假设 `data_type`。
- 本可批量处理的没有被拆成循环。
